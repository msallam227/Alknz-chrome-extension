MENA Component

Arabic Intelligence Engine Normalization & Extraction

The Arabic Engine improves capture quality in the MENA region by handling Arabic scripts, honorifics, name variants, RTL formatting, and organization naming patterns. This page documents the internal rules used to normalize Arabic entities before matching, enrichment, and upload.

Purpose

Why the Arabic Engine Exists

  • Arabic names have multiple valid spellings (e.g., محمد / Muhammed / Mohammed).
  • Honorifics and titles commonly appear inside names (دكتور، معالي، سمو).
  • Organizations use naming patterns not captured well by generic scrapers (القابضة، المالية، للاستثمار).
  • RTL text flow can break naïve parsing and tokenization.
  • Deduplication in MENA requires more than exact string matching.
Goal: Increase match accuracy and reduce duplicate profiles by normalizing Arabic before Smart Map and Deep AI.
Pipeline

Where Arabic Normalization Runs

Arabic processing can occur in two places:

Client-side (Extension)
  • Detect RTL and Arabic characters
  • Normalize obvious punctuation + whitespace
  • Pass “clean candidates” into lookup / Deep AI
Server-side (Backend)
  • Canonical name normalization (highest priority)
  • Alias generation for matching/dedup
  • Organization keyword classification
Rule of thumb: if normalization affects deduplication or stored identity fields, implement server-side.
Names

Arabic Person Name Normalization

The Arabic Engine converts noisy name strings into stable matching keys. It does not “change identity” — it creates consistent fields for lookup and dedup.

Normalize characters
  • Remove tashkeel/diacritics ( َ ِ ُ ّ ْ )
  • Normalize Alef variants: أ / إ / آ → ا
  • Normalize Yeh variants: ى → ي (optional)
  • Normalize Teh Marbuta: ة → ه (optional for matching only)
  • Normalize punctuation and extra spaces
Remove honorifics
Strip common tokens from the beginning of names for matching:
د. / دكتور / الدكتور / معالي / سعادة / سمو / الشيخ / الشيخة / الأستاذ / الأستاذة
Stored display_name can keep honorifics; matching_key should not.
Example
Input
د. مُحَمَّد بن عبدالعزيز آل سعود
Matching key (normalized)
محمد بن عبدالعزيز ال سعود
NOTE: “Al/Al-” romanization is handled separately. Arabic normalization is for Arabic script strings.
Organizations

Arabic Organization Normalization

MENA firms commonly include legal and business suffixes. The engine detects and classifies these tokens to improve matching (and reduce duplicates caused by suffix variation).

Common suffix tokens
القابضة • المالية • للاستثمار • للاستشارات • للتطوير • للتقنية • للتكنولوجيا • للتجارة • للمقاولات • للصناعة • للتمويل
These can be used for classification and alias generation.
Legal forms (often omitted)
شركة • شركة مساهمة • مساهمة مقفلة • ذات مسؤولية محدودة • ش.م.م • ش.م.ح • ش.ذ.م.م
For matching, generate aliases with and without legal forms.
Example
Input
شركة الراجحي المالية
Aliases for matching
الراجحي المالية
الراجحي
Matching

Deduplication & Matching Strategy

Exact string matching fails in Arabic contexts. The engine creates multiple matching keys to improve Smart Map lookup.

Recommended matching keys
{
  "display_name": "د. مُحَمَّد بن عبدالعزيز آل سعود",
  "name_key_ar": "محمد بن عبدالعزيز ال سعود",
  "name_key_ar_compact": "محمدبنعبدالعزيزالسعود",
  "aliases_ar": [
    "محمد عبدالعزيز ال سعود",
    "محمد بن عبدالعزيز"
  ],
  "romanized_aliases": [
    "Mohammed bin Abdulaziz Al Saud",
    "Muhammad bin Abdulaziz Al Saud"
  ]
}
For privacy and correctness: store display_name as seen; store keys/aliases for matching only.
UI

RTL Rendering & UI Handling

Arabic should be rendered RTL to avoid broken alignment and token confusion. The extension UI can toggle RTL mode when Arabic is detected.

Detection rule
If string contains Arabic Unicode range, enable RTL for that container: 
/[\u0600-\u06FF]/
Rendering rule
Apply a CSS class: 
.rtl { direction: rtl; font-family: 'Noto Sans Arabic'; }

Edge Cases

Common Arabic Edge Cases

1) Mixed Arabic + English
Some pages contain bilingual names. Keep both:
  • display_name_ar
  • display_name_en
  • matching keys for each
2) “Al” / “El” romanization drift
Arabic ال may appear as Al / El / Al- / El-. Generate multiple romanized aliases for lookup.
3) Tribe/family tokens
Tokens like آل / بن / بنت / أبو appear in names and should not be stripped. Only strip titles/honorifics.
4) Hamza & diacritics
Removing diacritics improves matching. Normalize hamza forms for stable keys.
Never “guess” identity. Normalization only improves string matching and reduces duplicates.
